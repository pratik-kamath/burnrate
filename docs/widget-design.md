# burnrate Widget — Design Spec

**Date:** 2026-06-02
**Status:** Proposed, pre-implementation. **Blocked on installing Xcode.**
**Goal:** Replace the floating overlay with a real **WidgetKit** widget (the kind in the macOS widget gallery — desktop / Notification Center, like Calendar & Weather) that shows Claude + Codex plan usage as ring gauges, fed by a small **background menu-bar host app**.

This supersedes the floating-overlay UI from `docs/design.md`. The data layer (`BurnrateCore`: providers, models, notifier) is **reused unchanged** except for small additions noted below.

---

## 1. Why this shape

A WidgetKit widget extension is **sandboxed** and **cannot**:
- read `~/.codex/auth.json` or the Claude Keychain item (outside its container), or
- run continuous timers / make freely-scheduled network calls.

So the widget cannot fetch usage itself. The standard pattern, which we adopt:

```
  ┌───────────────────────────┐         App Group container          ┌─────────────────────────┐
  │  burnrate (host app)       │   group.ai.multifactor.burnrate      │  burnrateWidget (.appex) │
  │  • menu-bar agent          │   ┌───────────────────────────┐      │  • TimelineProvider      │
  │  • fetches Claude + Codex  │──▶│ UsageSnapshot JSON (×2)     │─────▶│    reads snapshot        │
  │  • writes shared snapshot  │   └───────────────────────────┘      │  • renders rings         │
  │  • WidgetCenter.reload()   │──────────── reload signal ──────────▶│  • system + pushed reload│
  └───────────────────────────┘                                       └─────────────────────────┘
```

The host stays running and pushes `WidgetCenter.shared.reloadAllTimelines()` whenever it fetches fresh data, so the widget updates roughly as often as the host pushes (subject to WidgetKit throttling) rather than only on the system's idle budget.

---

## 2. Hard requirements (the prerequisites)

1. **Full Xcode** (App Store). WidgetKit app extensions, App Groups, and code signing cannot be produced by SwiftPM + Command Line Tools.
2. **Apple ID signed into Xcode** (Settings → Accounts). A **free personal team** is sufficient for installing on your own Mac. App Groups + widget extensions require signing with a team (ad-hoc `-` signing won't do).
3. **XcodeGen** (`brew install xcodegen`) — generates `burnrate.xcodeproj` from a checked-in `project.yml`, so the project is reproducible and reviewable instead of a hand-edited `.pbxproj`.

---

## 3. Targets & project layout

Generated by XcodeGen from `project.yml`:

| Target | Type | Role |
|---|---|---|
| **BurnrateCore** | local SwiftPM package (existing) | Models, providers, notifier, color logic. Linked by both targets below. **UI-free.** |
| **burnrate** | macOS app (`LSUIElement`, menu-bar agent) | Fetches usage, writes shared snapshot, pushes widget reloads, fires notifications, menu-bar UI. Embeds the widget. |
| **burnrateWidget** | Widget extension (`.appex`) | `TimelineProvider` + SwiftUI ring rendering, reads shared snapshot. |

Both `burnrate` and `burnrateWidget` carry the **App Group** entitlement `group.ai.multifactor.burnrate`.

```

## 4. Additions to BurnrateCore (small)

- **`Codable` conformance** on `Provider`, `UsageStatus`, `WindowUsage`, `UsageSnapshot` (so snapshots serialize to the shared container).
- **`SharedSnapshotStore`** — reads/writes the two latest snapshots to `UserDefaults(suiteName: "group.ai.multifactor.burnrate")` as JSON under keys `snapshot.claude` / `snapshot.codex`. Used by the host (write) and the widget (read). Pure, no UI.

The existing `ClaudeUsageProvider`, `CodexUsageResponse`, `BackoffPolicy`, `MilestoneNotifier`, `UsageLevel` are reused as-is. (`CodexLiveProvider` stays in the app target — it reads `~/.codex/auth.json`, which only the full app can do.)

---

## 5. Host app (`burnrate`) behavior

- **Menu-bar agent**: `LSUIElement = true`, an `NSStatusItem` with a small icon and menu: **Refresh Now**, **Notifications** (toggle), **Open Widget Gallery help**, **Launch at Login** (via `SMAppService`), **Quit**.
- **Fetch loop** (`UsageFetcher`): Codex live every 60s, Claude every 5 min with `BackoffPolicy` (identical cadence to the overlay). On each result: write to `SharedSnapshotStore`, run `MilestoneNotifier`, and call `WidgetCenter.shared.reloadAllTimelines()`.
- **Notifications** remain the host's job (the widget never notifies).
- No window/overlay — the menu-bar item is the only chrome.

---

## 6. Widget extension (`burnrateWidget`)

- **Sizes:** `.systemSmall` (two compact rings, 5-hour %) and `.systemMedium` (two rings + weekly % + reset countdown text). `supportedFamilies` limited to these two.
- **`TimelineProvider`:**
  - `placeholder` / `snapshot`: synthetic sample data for the gallery preview.
  - `getTimeline`: read both snapshots from `SharedSnapshotStore`; emit a single entry for "now" with `reloadPolicy: .after(Date().addingTimeInterval(900))` (15-min safety net). The host's `reloadAllTimelines()` is the primary freshness driver; this policy is just a fallback for when the host isn't running.
- **Rendering:** `WidgetRingView` adapted from the overlay's `RingView` (same `UsageLevel` color thresholds, terracotta/green tints). No hover in widgets, so medium shows weekly + reset inline; small shows just the two 5-hour rings with tiny labels.
- **Unavailable/stale:** if a snapshot is missing or `status == .unavailable`, that ring shows `—`. If the host hasn't written in a while, the widget simply shows the last snapshot (no live "as of" needed since the host pushes reloads when running).

---

## 7. Signing & build

- `project.yml` sets `CODE_SIGN_STYLE: Automatic` and a `DEVELOPMENT_TEAM` (your personal team id — captured once Xcode is set up; or selected in Xcode's GUI on first build).
- App Group capability declared in both entitlements files and in `project.yml`.
- Build/run: open the generated project in Xcode and **Run** the `burnrate` scheme once (installs the host app + embedded widget). Then add the widget from the gallery (right-click desktop → **Edit Widgets** → burnrate, or Notification Center). CLI build: `xcodebuild -scheme burnrate -configuration Release`.

---

## 8. Freshness expectations (honest)

- **Host running:** widget updates shortly after each host fetch (host pushes reloads). Practically near-live, modulo WidgetKit's reload throttling.
- **Host not running:** widget shows the last-written snapshot and only refreshes on the 15-min fallback policy / system budget — and Claude/Codex data won't advance at all (nothing is fetching). So the host should **Launch at Login**.

---

## 9. Open questions / risks

- **App Group + free team:** App Groups are supported on free personal teams for local install, but provisioning occasionally needs a one-time Xcode GUI nudge (select team, let it create the profile). The plan will call this out as a manual step.
- **Widget reload throttling:** Apple budgets widget reloads; under heavy use the widget may lag the menu-bar truth by a few minutes even with pushed reloads. Acceptable for a usage gauge.
- **`DEVELOPMENT_TEAM` in `project.yml`:** committing a team id is harmless (it's not a secret), but we can also leave it blank and select in Xcode to keep the file generic. Decide during implementation.
- **Keep a status readout in the menu bar?** Optional: show the higher of the two 5-hour %s as the status-item title so you get a glance without the widget. Easy add; flag for v2.
